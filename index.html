Cross Platform

Nape is written in cross-platform Haxe and is compiled to .swc libraries for ActionScript3 users enabling use across Flash Player, Desktop, HTML5, Android, iOS...

Fast

Nape is fast. Significantly faster than both Box2DAS3 and Box2DAlchemy, it is also very memory concious: leaving flat memory graphs in its wake with no GC pauses or memory thrashing.

With the exclusion of a single object (BitmapDebug), Nape does not make use of any Alchemy opcodes, and so can be used royalty free in combination with Stage3D APIs in Flash

Friendly API

Nape's API is very flexible and concise, permitting a wide array of manipulations to be made which are handled seamlessly in a lazy system for best performance. 

// Example creating a Box rigid body and adding to a newly created space.
// No utility methods beyond Polygon.box are used, this is as verbose as we can get.
var space = new Space();
var boxBody = new Body();
boxBody.shapes.add(new Polygon(Polygon.box(width, height)));
boxBody.space = space;

// Modifying velocity of body
body.velocity.x += 100;

Nape uses SI units; but with metres replaced by... pixels! It is mybelief - and few have argued otherwise - that pixels is always the most natural unit to use in 2D games, and by extension for 2D physics.

Checked API

Nape is built in two distinct modes, in the development mode, the Nape API will actively check that you are using it correctly, and inform you when you are misusing it. This is undoubtedly very helpful when approaching a large library.

The other build mode for Nape is the release mode which is used for benchmarking and final releases of games which strips out all such checks for a faster, lighter build.

For ActionScript3 users, these modes are achieved through distinct .swc libraries whilst Haxe users will switch modes with a compiler flag.

Rigid Bodies

  • Compound rigid Body's formed of Circle and convex Polygon shape primitives. Polygons can be given in any winding.
  • Stable and accurate collisions with block solving of contact patches.
  • Elastic collisions with static, dynamic and rolling frictions controlled by Material objects assigned to Shapes.
  • Continuous Collision Detection (CCD) to avoid tunneling at high speeds. Enabled always for static and kinematic collisions and sensors, and on demand for dynamic<->dynamic collisions and sensors by setting isBullet property on Body.
  • And lots more!

body.wake()

  • In Nape, you will never (*) need to manually wake up a Body, or call any method that recomputes certain values at any time. Everything is handled completely by Nape, and is done lazily.
  • However, I understand that sometimes... you really want to be able to manually force something to go to sleep. And this is provided for those who require it by the nape-hacks module's ForcedSleep object.
(*) The only time you will ever need to manually wake up a Body, is if you yourself manually forced it to sleep

Buoyancy

  • Rigid body Shapes marked as fluidEnabled are able to interact with buoyancy and fluid drag in place of standard collisions.
  • Fluid enabled Shapes can belong to any Body, even dynamic ones permitting fun, crazy game mechanics and flexibility.
  • Fluid buoyancy computed accurately, with fluid drag effected by shape profiles (An arrow pointing forwards moves through fluid easier than against its direction).

Constraints

  • Many in-built Constraint types: PivotJoint, DistanceJoint, AngleJoint, MotorJoint, LineJoint, WeldJoint. All in-built constraints are single-purpose and designed to be composed.
  • All constraints can be made stiff or elastic with parameters to control frequency and damping of elastic constraints. (*Excludes velocity only constraints like MotorJoint as there is no concept of elasticity or stiffness).
  • All constraints support parameters to provide limits to how much error may be tolerated before a constraint is broken, or in the case of elastic constraints, control how much force is permitted to be used to try and solve the constraint, and how much positional error should be resolved per second.
  • UserConstraint API providing higher-level access to creating custom constraints. UserConstraints can still be made stiff or elastic, with all above mentioned features present with no user effort.
  • Through use of the nape-symbolic module, make use of the SymbolicConstraint API using a high-level DSL (Domain Specific Language) to easily, and rapidly prototype new constraints.

High Level Callbacks

  • Callback system operates on-top of CbType's which are assigned to Shapes, Bodies, Compounds and Constraints, and Listeners that listen over inclusion/exclusion sets of CbTypes.
  • Callback system is highly modular, objects can be given as many CbTypes as you wish to add behaviour to the simulation at any time in development without modifying lots of other code.
  • Interaction type callbacks operate between pairs of objects, whether they be Shapes Bodies or Compounds. Events may be raised based on the CbTypes so that a single callback occurs when a Body of a particular type collides for the first time with a Compound of a particular type.
  • Lower level PreListeners give access to contact information before collisions are resolved permitting case-by-case decisions on whether a collision will go ahead (One-way platforms for instance) based on contact data, as well as permitting setting precise friction and elasticity coeffecients on the contact patch.
  • System is implemented so as to be super effecient without significant overhead
 
var Breakable = new CbType();
var IgnoreThese = new CbType();

constraint.cbTypes.add(Breakable);
constraint.cbTypes.add(IgnoreThese);

// Listen for 'any' constraint that does not
// have the IgnoreThese CbType.
var wakeListener = new ConstraintListener(
    CbEvent.WAKE,
    CbType.ANY_CONSTRAINT.excludes(IgnoreThese),
    wakeHandler
);
wakeListener.space = space;

// Listen for only those constraints having
// the Breakable CbType.
var breakListener = new ConstraintListener(
    CbEvent.BREAK,
    Breakable,
    breakHandler
);
breakListener.space = space;

Fast, Stable Geometric Utilities

  • Nape's GeomPoly API provides the majority of such utilities with methods such as to perform convex decompositions: turning concave polygons which Nape does not support for physics, into convex partitions which can be used in Nape. Among other utilities, GeomPoly object also provides a method to cut a polygon into sub-polygons based on a given line segment. As well as simplification to reduce the number of vertices.
  • Nape's MarchingSquares object provides methods to perform surface extractions of a scalar field into weakly-simply polygons that may be directly fed into the convex decomposition method. Perfect for converting bitmaps into polygons based on alpha tolerances or generating procedural objects with Perlin Noise
  • Nape's Geom object provides an interface onto Nape's collision detection routines, with methods such as to compute the signed distance between two Shapes.

Build types

Nape exists in 3 different build types:

Development:
To be used during development of games. This build contains extra logic that checks you are using the Nape API correctly and is invaluable in reducing development time and ensuring you have no bugs with respect to Nape.
Release:
Release builds remove all such API checks so that the best performance may be found. This built type should be used for FINAL builds and performance testing only.
Debug:
Debug builds should not be used in general. This build type is very slow; containing tens of thousands of internal assertions whose purpose is to identify bugs internal to Nape.

Haxe

Haxe users can make use of Nape via haxelib 

// For core library.
haxelib install nape

// For hacks module.
haxelib install nape-hacks

// For symbolic module.
haxelib install nape-symbolic

Build types in Haxe are controlled by use of compiler flags.

Development:
This is the default build type.
Release:
Use additional flags -D NAPE_RELEASE_BUILD
Debug:
Use additional flags -D NAPE_ASSERT

AS3

AS3 users must make use of Nape through the compiled .swc libraries, one per each build type:

When making use of modules, you must ensure the core library is linked first.

Core Library:

nape-dev.swc nape-release.swc nape-debug.swc

Hacks module:

nape-hacks.swc

Symbolic module:

nape-symbolic.swc
